# Core Concepts Overview

The HPC Neura Execution Engine is built on a sophisticated multi-level architecture that separates concerns between different stakeholders while maintaining flexibility and security. This overview introduces the fundamental concepts and terminology that form the foundation of the system.

## Architecture Philosophy

<Card>
  <CardHeader>
    <CardTitle>Separation of Concerns</CardTitle>
  </CardHeader>
  <div style={{ padding: '1rem' }}>
    <p>The HPC Neura Execution Engine implements a four-level architecture where each level has distinct responsibilities:</p>
    
    <Features>
      <Feature title="Template Definition" icon="/img/icons/code.svg">
        L1 developers create reusable element templates
      </Feature>
      <Feature title="Workflow Design" icon="/img/icons/network.svg">
        L2 developers build flows using elements
      </Feature>
      <Feature title="User Interaction" icon="/img/icons/frontend.svg">
        L3 users execute flows with their data
      </Feature>
      <Feature title="Runtime Execution" icon="/img/icons/hpc.svg">
        L4 runtime processes flows and generates results
      </Feature>
    </Features>
  </div>
</Card>

## Key Terminology

Understanding the terminology is essential for working with the HPC Neura Execution Engine:

<Tabs>
  <TabItem value="components" label="Core Components" default>
    <div style={{ marginTop: '1rem' }}>
      <Card>
        <CardHeader>
          <CardTitle>Elements</CardTitle>
        </CardHeader>
        <div style={{ padding: '1rem' }}>
          <p><strong>Definition:</strong> Predefined templates created by L1 developers that encapsulate specific functionality.</p>
          <p><strong>Storage:</strong> Stored in the <code>element_data</code> table.</p>
          <p><strong>Purpose:</strong> Provide reusable, secure building blocks for flows.</p>
          <p><strong>Examples:</strong> LLM Text Generator, Data Selector, API Connector</p>
        </div>
      </Card>
      <br></br>
      <Card style={{ marginTop: '1rem' }}>
        <CardHeader>
          <CardTitle>Nodes</CardTitle>
        </CardHeader>
        <div style={{ padding: '1rem' }}>
          <p><strong>Definition:</strong> User-created instances of elements within a flow.</p>
          <p><strong>Created By:</strong> L2 flow developers from element templates.</p>
          <p><strong>Purpose:</strong> Perform specific operations within a workflow.</p>
          <p><strong>Customization:</strong> Configured according to element hyperparameters.</p>
        </div>
      </Card>
      <br></br>
      <Card style={{ marginTop: '1rem' }}>
        <CardHeader>
          <CardTitle>Flows</CardTitle>
        </CardHeader>
        <div style={{ padding: '1rem' }}>
          <p><strong>Definition:</strong> Complete workflows consisting of connected nodes.</p>
          <p><strong>Structure:</strong> Directed Acyclic Graphs (DAGs).</p>
          <p><strong>Purpose:</strong> Define end-to-end processes for specific tasks.</p>
          <p><strong>Components:</strong> Nodes, connections, inputs, outputs, metadata.</p>
        </div>
      </Card>
      <br></br>
      <Card style={{ marginTop: '1rem' }}>
        <CardHeader>
          <CardTitle>Connections</CardTitle>
        </CardHeader>
        <div style={{ padding: '1rem' }}>
          <p><strong>Definition:</strong> Data pathways between nodes.</p>
          <p><strong>Format:</strong> <code>node_id:variable_name</code></p>
          <p><strong>Purpose:</strong> Define how data flows through the workflow.</p>
          <p><strong>Types:</strong> Simple (all data) or field-specific mappings.</p>
        </div>
      </Card>
    </div>
  </TabItem>
  
  <TabItem value="concepts" label="System Concepts">
    <div style={{ marginTop: '1rem' }}>
      <Card>
        <CardHeader>
          <CardTitle>Hyperparameters</CardTitle>
        </CardHeader>
        <div style={{ padding: '1rem' }}>
          <p><strong>Definition:</strong> Rules that control how elements can be customized.</p>
          <p><strong>Access Levels:</strong></p>
          <ul>
            <li><code>fixed</code> - Cannot be modified</li>
            <li><code>edit</code> - Full control within constraints</li>
            <li><code>append</code> - Can add but not remove</li>
            <li><code>hidden</code> - Present but not visible</li>
          </ul>
          <p><strong>Purpose:</strong> Balance flexibility with security and correctness.</p>
        </div>
      </Card>
      <br></br>
      <Card style={{ marginTop: '1rem' }}>
        <CardHeader>
          <CardTitle>Schemas</CardTitle>
        </CardHeader>
        <div style={{ padding: '1rem' }}>
          <p><strong>Input Schema:</strong> Defines expected input structure and types.</p>
          <p><strong>Output Schema:</strong> Specifies what a node produces.</p>
          <p><strong>Parameter Schema:</strong> Constrains configuration options.</p>
          <p><strong>Purpose:</strong> Ensure type safety and data compatibility.</p>
        </div>
      </Card>
      <br></br>
      <Card style={{ marginTop: '1rem' }}>
        <CardHeader>
          <CardTitle>Execution Context</CardTitle>
        </CardHeader>
        <div style={{ padding: '1rem' }}>
          <p><strong>Definition:</strong> Runtime information about the user and environment.</p>
          <p><strong>Includes:</strong> User details, session info, feature flags.</p>
          <p><strong>Access:</strong> Available to nodes via metadata elements.</p>
          <p><strong>Purpose:</strong> Provide context-aware execution.</p>
        </div>
      </Card>
    </div>
  </TabItem>
</Tabs>

## Execution Model

The HPC Neura Execution Engine follows a sophisticated execution model:

<Features>
  <Feature title="Dependency Resolution" icon="/img/icons/ai-workflow.svg">
    Nodes execute when all dependencies are satisfied
  </Feature>
  <Feature title="Parallel Processing" icon="/img/icons/hpc.svg">
    Independent nodes run concurrently for performance
  </Feature>
  <Feature title="Event Streaming" icon="/img/icons/network.svg">
    Real-time updates during execution
  </Feature>
</Features>

### Execution Flow

<CollapsibleCodeBlock
  title="Execution Lifecycle"
  description="High-level view of how flows execute"
  language="text"
>
{`1. Flow Initialization
   - Load flow definition
   - Validate structure
   - Prepare execution context

2. Input Processing
   - Receive L3 user inputs
   - Map to node inputs
   - Validate against schemas

3. Node Execution
   - Resolve dependencies
   - Execute nodes in order
   - Handle parallel paths

4. Data Transformation
   - Process node outputs
   - Route through connections
   - Transform data types

5. Result Generation
   - Collect end node outputs
   - Format response
   - Return to user`}
</CollapsibleCodeBlock>

## Data Flow Principles

<Card>
  <CardHeader>
    <CardTitle>How Data Moves Through the System</CardTitle>
  </CardHeader>
  <div style={{ padding: '1rem' }}>
    <ol>
      <li><strong>Input Stage:</strong> L3 users provide data mapped to specific nodes</li>
      <li><strong>Processing Stage:</strong> Nodes transform data according to their logic</li>
      <li><strong>Transfer Stage:</strong> Connections route data between nodes</li>
      <li><strong>Output Stage:</strong> End nodes produce final results</li>
    </ol>
    
    <Callout type="info">
      Data is strongly typed and validated at each stage to ensure correctness and security.
    </Callout>
  </div>
</Card>

## Summary

The HPC Neura Execution Engine provides a powerful, flexible, and secure platform for building and executing complex workflows. Its multi-level architecture ensures:

- **Separation of Concerns:** Each level has clear responsibilities
- **Security:** Sensitive operations are hidden and protected
- **Flexibility:** Customization within defined boundaries
- **Scalability:** Efficient execution with parallel processing
- **Type Safety:** Strong typing and validation throughout
